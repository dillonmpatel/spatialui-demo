3D as Code SpatialUI Roadmap

Text became the universal interface for software.
3D is the universal interface for space.

Code

Medium: Text — instructions, logic, programs
Properties: Inspectable, editable, versionable, composable
AI unlock: LLM Code — the LLM sees your repo

3D

Medium: Geometry — shapes, assemblies, worlds
Properties: Inspectable, editable, versionable, composable
AI unlock: LLM 3D — the LLM sees in 3D, and you point back
1

3D is the spatial equivalent of code

  • Code is the interface between humans and AI for software. 3D plays the same role for physical and virtual worlds — to generate, edit, simulate, and communicate between both.
  • This needs to be a two-way interaction. The LLM should be able to annotate, highlight, and manipulate 3D views. The user should be able to select, point, and manipulate back. The 3D viewer is a shared workspace you and the LLM can look at together.
  • The other approach is asking the model to be the program instead of write one. Generative 3D ignores what makes code useful — you can't inspect it on a granular level, direct it to another tool, or provide specific feedback.
2

LLMs may already have the foundation

  • LLM Code showed that tool access can unlock latent capability. LLMs can already reason about code — bash and grep gave it the way in. I ran a leading LLM on GeoGramBench — it scored 97.6% (488/500). LLMs can translate procedural descriptions and geometric understanding. It just needs the right tools.
  • I use LLM Code daily in hardware R&D at Apple. It helps with isolated software problems, but I'm hitting the ceiling of what isolated tools can do. The next step is giving an LLM eyes for geometry — the spatial equivalent of bash and grep.
  • Product design and engineering is the right place to start. It has well-defined workflows, structured data (parametric CAD and GD&T), and tight feedback loops. Broader applications like architecture, robotics, and digital twins will follow naturally.
3

What it would look like

  • LLM 3D, but for physical products. Open a 3D assembly. the LLM sees the geometry, traverses the feature tree, measures distances, and cross-references with simulation, PCB, and supply chain data.
  • Try the demo: click "Case 1: ID wants 30% smaller" on the home page. the LLM identifies the thermal impact, flags parts that no longer fit, checks the power envelope, and estimates the cost delta — pulling from CAD, simulation, and ERP in one response. That cross-domain reasoning currently takes a room full of specialists.
  • 3D today is fragmented. Dozens of file formats, incompatible renderers, tools that cost thousands and take years to learn. Something like LLM 3D could make spatial design accessible down to the consumer level — people could interact with natural language and work with 3D design tools in ways that have been completely inaccessible to them until now.
  • For the detailed product roadmap and phasing, see the next page.
4

The honest gaps

  • This isn't ready to ship tomorrow. GeoGramBench tests procedural geometry, not real CAD files. Metric measurement on production assemblies needs tool augmentation and likely vision fine-tuning. Open questions around CAD encoding, token-efficient geometry, and MCP schema design. Real gaps, but well-scoped and parallelizable.
  • Getting this right requires the right R&D culture and trust. Taking research into active demos and dogfooding real feedback is how this kind of work moves. But when tools touch physical systems, the reliability bar is completely different than software.
  • My experience on Face ID shaped how I think about this. We ran data collection, training, and hardware development internally — because the on-device model had to work for everyone, everywhere.
